# AGENTS.md — Правила разработки Smartmessage.kz

Этот файл — обязательный к исполнению свод правил для всех ИИ-агентов и людей, работающих в репозитории. Он применяется ко всему репозиторию, если более глубокий `AGENTS.md` не переопределяет отдельные секции.

Проект: B2B SaaS-платформа сквозной экосистемы (ИИ-таргет → CRM + чат-бот → дожим базы через WhatsApp-рассылки → обучение → подписка $200–1000). Деньги, рассылки, база клиентов, ИИ-агенты — ошибки стоят дорого. Правила строгое соответствие этому контексту.

## 0. Технологический стек (фиксированный)

Одобрен владельцем. Не менять без явного разрешения.

| Слой | Технология | Назначение |
|------|------------|------------|

## 4. Task Mode (классификация перед правками)

Перед редактированием классифицировать задачу и масштабировать процесс:

- **`Direct`**: косметика, копирайт, отступы, стили, комментарии, очевидные локальные правки без изменения поведения. Изучить затронутый файл и соседние использования, сделать минимальное связное изменение, запустить узкую валидацию если дёшево.
- **`Investigation`**: диагностика/дебаг, когда корневая причина или путь сбоя ещё не ясны. Воспроизвести или проследить путь сбоя, вертикальное и горизонтальное исследование перед патчем, найти владеющий слой, остановиться и переформулировать задачу если две попытки не двигают сигнал.
- **`TDD-first`** (поведение/логика/контракты/аутентификация/права/персистентность/валидация/состояние/конкурентность/нетривиальный UI): предпочтительно начинать с самого ценного падающего теста. От пользовательского или контрактного поведения, узкие unit-тесты для чистых правил с высоким ветвлением. Строгий цикл: красный → минимальная реализация → зелёный → следующий случай.

**Для этого проекта TDD-first обязателен** для: любого кода рассылок, биллинга/подписок, аутентификации и прав, интеграций с WhatsApp/Instagram/Facebook/TikTok API, ИИ-агентов, маршрутизации лидов.

Не создавать церемонии для простых задач. Реализовывать напрямую, если правильный ход очевиден и низкорисков.

## 6. Vertical And Horizontal Research

Перед исправлением нетривиального поведения инспектировать и runtime-путь, и соседние системы.

**Вертикальное** (по пути исполнения):
- Фронт: UI → маршрут/guard/layout → page/container → hook/handler → Server Action/API → persistence.
- Воркер: trigger → BullMQ job → воркер → домен → persistence/внешний сервис.
- Async: триггер → очередь/job → retry/идемпотентность → side effect → видимость статуса/ошибки.

**Горизонтальное** (смежные поверхности, должны оставаться согласоваными):
- соседние роуты, похожие компоненты, связанные хуки, общие сервисы, схемы, сериализаторы, тесты, существующие паттерны;
- состояния: loading, empty, error, success, disabled, optimistic, retry, stale-cache;
- продюсер и потребитель контракта;
- пути чтения и записи для изменений персистентности.

Достаточно исследования, чтобы найти владеющий слой. Не превращать исследование в блуждание.

## 7. Root Cause Discipline

- Не латать симптомы до понимания пути сбоя.
- Чинить владеющий слой, а не ближайший видимый симптом.
- Если баг в дочернем компоненте/hook/функции — сначала инспектировать родительский/владеющий слой.
- Отвергать дочерние компенсации, defensive state repair, дублирование логики, локальные ветвления, merely прячущие upstream-ошибку.
- One-file fix для кросс-слойного поведения подозрителен, пока не доказано обратное.
- Не сохранять сломанное решение через guards/flags/wrappers/дублирование.
- Если минимальный diff и правильный diff расходятся — выбирать правильный с наименьшим системным следом.
- Изменение не минимально, если оно делает код менее понятным завтра.
- Если требуется ре-архитектура — озвучить скоуп, риски, обратную совместимость, порядок rollout.

> Опыт донора: `retry_onfail` пересоздавал сокет, но не повторял отправку сообщения → терялись сообщения. Не повторять.

## 8. Change-Surface Triggers

При касании границы — инспектировать и выравнивать напрямую связанный код:

- **Общие контракты/схемы**: валидировать продюсера и потребителя.
- **Маршруты, guards, редиректы, layouts**: инспектировать защищённые/публичные потоки, родительскую оркестрацию, side effects навигации.
- **Мутации/запросы (Server Actions, React Query)**: инспектировать ключи, инвалидацию, состояния loading/empty/error/success/optimistic/stale.
- **Схема/персистентность**: форму контракта, сериализаторы, миграции, сгенерированный клиент, пути чтения/записи.
- **Аутентификация/права**: guards, loaders, форму сессии, backend-enforcement, пользовательские состояния.
- **Async-воркеры (рассылки/очереди)**: retry, идемпотентность, порядок, отмена, видимость сбоя, rate-limit.
- **Интеграции внешних API (WA/IG/FB/TikTok/OpenAI)**: обработку ошибок, retry/backoff, лимиты, секреты, версионирование API.
- **Юридический/биллинг/приватность/безопасность**: сохранять продуктовый контракт, флаговать неоднозначность.

## 9. Minimal Sufficient Change

- Целиться в минимальное связное изменение, полностью решающее проблему на владеющем слое.
- Минимальное = минимальная площадь поверхности, движущихся частей и количества абстракций, а не минимальный diff любой ценой.
- Плоские простые реализации предпочтительнее лишних слоёв, папок, паттернов.
- Локальная ясность предпочтительнее умного переиспользования.
- Декаплинг предпочтительнее DRY. Небольшое намеренное дублирование лучше неправильной общей абстракции.
- Не добавлять абстракции/hook'и/сервисы/wrapper'ы/папки/скрипты/генераторы, если они не убирают реальную текущую сложность.
- Делить код только когда это явно улучшает понимание или изолирует ответственность.
- Не строить framework-like архитектуру под небольшие фичи.

## 10. Documentation Discipline

- Код — первоисточник истины для деталей реализации.
- `docs/` должен фиксировать **долговечный контекст**: архитектуру, потоки данных, операционные ограничения, runbook'и, кейвы, нетривиальные решения.
- Не зеркалировать структуру кода в док, не вести исчерпывающий инвентарь файлов вручную, не создавать doc-churn для тривиальных рефакторингов.
- Обновлять доки, когда изменение materially затрагивает архитектуру, setup, операции, контракты, пользовательские потоки, важные инженерные решения.
- После реализации проверить, не нужно ли добавить/выровнять долговечное знание.
- Если выравнивание вышедшего из скоупа doc-drift осталось — явно отметить.
- Любое тектоническое изменение (WA-процессы, биллинг, auth, БД, очередь) фиксируется микро-ADR в `docs/adr/`. ИИ-агент не пересматривает решение из ADR без явного указания владельца. Мелкие решения в ADR не пишем (анти-doc-churn).

## 11. Testing And Validation (обязательно для этого проекта)

Подробности — в `docs/TESTING.md`. Здесь — обязательные минимумы:

- **Каждая нетривиальная правка сопровождается тестами.** TDD-first для: рассылок, биллинга, auth/прав, внешних интеграций, ИИ-агентов, маршрутизации лидов.
- Запускать минимальную осмысленную валидацию, покрывающую изменённую поверхность. Сначала дешёвые gate'ы: точечные тесты → typecheck → lint → build → целевые скрипты → более широкие сьюты при необходимости.
- Использовать существующую тест-инфраструктуру. Не выдумывать более тяжёлый тест-слой без явного обоснования.
- Валидировать после реализации и до закрытия задачи.
- При изменении контрактов/общих схем — валидировать обе стороны (продюсер + потребитель).
- Трактовать как провал валидации: non-zero exit, runtime-ошибки, unhandled rejections, failed assertions, type errors, lint errors, build failures, timeouts.
- Не объявлять успех по прокси-метрикам. Зелёные тесты/lint/typecheck недостаточны, если первичный пользовательский сигнал всё ещё сломан.
- Если проверены только вторичные сигналы — отчитаться как «частично валидировано».
- Если валидацию нельзя запустить — сказать почему и назвать лучший доступный заместитель.

### Системы с деньгами/рассылками/правами — повышенные требования

Для кода, затрагивающего биллинг, списания, рассылки, права доступа, интеграции внешних API — обязательны:
- интеграционный тест с моком внешнего сервиса (WA/IG/FB/TikTok/OpenAI);
- проверка идемпотентности (повторный запуск того же задания не дублирует side effect);
- проверка retry/backoff на временных сбоях;
- для рассылок — проверка rate-limit/jitter/ротации аккаунтов;
- для биллинга — тест на корректность списания, на отказ/возврат, на граничные суммы.

### Жёсткие тест-гейты

- **Anti-weakening rule:** падающий тест чинится **только исправлением основного кода**. Категорически запрещено: `test.skip`, `it.only`, `xit`, комментирование/удаление ассертов, искусственное занижение порога покрытия ради зелёного пайплайна.
- **Двухслойный барьер контроля:**
  1. *Локально:* git-хук (Husky) на pre-push автоматически запускает `pnpm build && pnpm test`.
  2. *Независимо:* финальный гейт — серверный CI (GitHub Actions), который агент не может редактировать; падает при появлении `.skip`/`.only`/`xit` в диффе тестов и при просадке покрытия критических зон.
- **Test-evidence ledger:** каждая фича/задача требует фиксации артефактов прогона в `docs/qa-runs/` (см. `docs/qa-runs/_TEMPLATE.md`). Нет заполненного отчёта — задача не «Done».

## 13. Безопасность и секреты

- **Секреты — только в переменных окружения**, никогда в коде/коммите. (Донор хранил admin_api_key и access_token в БД/query — не повторять.)
- Токены/`access_token`/`instance_id` — **в заголовках Authorization или cookie**, не в URL query (query попадает в логи прокси).
- SQL — только через параметризованные запросы Prisma. Никакой интерполяции строк в SQL. (Донор имел SQL-injection поверхность.)
- CORS — белый список origin, не `*`. (Донор имел `origin:'*'`.)
- Body-parser лимиты — разумные (не 50mb без причины).
- Валидация ввода на каждой границе (zod-схема на API/Server Action).
- Идемпотентность на мутациях с side effects (платежи, отправки) через idempotency-key.
- **Вебхуки платёжного провайдера (MoR, см. `docs/adr/0002-payment-gateway.md`) маппятся на тенанта только через метаданные.** При создании Checkout Session обязательно передавать `metadata: { teamId, subscriptionId }` + `client_reference_id`. В обработчике вебхука искать `Subscription` по `providerSubId` (`@unique`), брать оттуда `teamId`; рядом хранить `paymentProvider` (какой именно MoR). В payload провайдера `teamId` нет — не искать его там. Отсутствие маппинга = явная ошибка, не silent loss.
- Не открывать endpoints без аутентификации/авторизации, кроме явно публичных.
- Не выводить stack trace'ы и внутренние ошибки наружу в production.

## 14. Рассылки и WhatsApp — особые правила

Эти правила выведены из анализа донора (см. `docs/REUSE_DONOR.md`).

### Обязательно
- **Очередь — только BullMQ/Redis.** Никакого in-memory `bulks = {}` и `node-cron` polling как единственной очереди. Прогресс персистится в БД/Redis, не теряется при рестарте.
- **Идемпотентность отправки.** Один и тот же message-id/contact/campaign-id не уходит дважды даже при retry/рестарте воркера. Использовать unique-job-id + дедупликацию в БД.
- **Single-process ownership WA-сокета.** Жизненный цикл Baileys-сокета одного `instanceId` принадлежит ровно одному физическому процессу (sticky ownership) на всём протяжении — это long-lived stateful ресурс, короткие блокировки с TTL бессильны (сокет-шторм, «conflict: replaced»). WA-сокеты — в выделенном процессе (или шардированном наборе с leader-election на шарде); stateless-задачи масштабируются в общем пуле BullMQ. **Job отправки не открывает сокет** — он направляет команду процессу-владельцу через Redis-реестр `instanceId → workerId`. Распределённая блокировка — только вспомогательное средство при миграции владельца, не основной механизм.
- **Глобальный rate-limit на уровне WA-аккаунта** через BullMQ `limiter: { max, duration }` + token-bucket. Донор имел только per-кампанию паузы → N кампаний = xN скорости → бан.
- **Троттлинг с jitter**: random delay между сообщениями (per-кампания `min_delay`/`max_delay`) **плюс** ротация аккаунтов (`next_account`).
- **Retry с backoff на временные сбои** (`session_error`, `timeout`, `Bad MAC`) — повторить **отправку того же сообщения** после переподключения, а не просто пересоздать сокет. (Донор терял сообщения.)
- **Circuit-breaker**: при `>= N` технических сбоев подряд кампания авто-паузится. (Донор: `BULK_TECHNICAL_FAILURE_LIMIT=3`.)
- **Терминальные/временные статусы блокировки WA** (не путать с обычным `logged_out`):
  - `banned` — перманентный бан номера от Meta: терминальный, аккаунт навсегда исключается из ротации, авто-реконнект **не** пытаемся, алерт владельцу команды + `AuditLog`.
  - `restricted` — временная блокировка/rate-limit: пауза до `restrictedUntil`, затем auto-reconnect и возврат в ротацию.
  - `classifySendError` (`packages/shared`) обязана отличать ответ о бане/ограничении от обычного `session_error`.
- **Spintax per-message** на всех полях шаблона для анти-дубликата текста.
- **Валидация номеров `onWhatsApp`** с enum-статусами (null/confirmed/not_on_whatsapp/in_progress/error) как асинхронная job-очередь.
- **Нормализация номеров и JID** — pure-функции в `packages/shared`. Критично для доставки.
- **Graceful shutdown**: ловить SIGTERM/SIGINT, дожидаться текущих job'ов, корректно закрывать WA-сокеты. (Донор не имел.)
- **Статусы отправок** (queued/sent/delivered/read/failed) хранить в БД и агрегировать, не только пушить в webhook.
- **Агрегаты для UI — из `Stats`**, не из `MessageLog`. Прямой `COUNT/GROUP BY` по живой `MessageLog` в пользовательских потоках запрещён (см. `docs/ARCHITECTURE.md` §7).
- **Классификация ошибок** `get_send_error_type`-стиль: `media_error`/`session_error`/`send_error` → разные стратегии retry.

### Категорически нельзя
- Использовать `forEach(async ...)` как «параллельность» без await и лимитов. Только `for...of` или BullMQ worker concurrency.
- Хранить прогресс рассылки только в RAM процесса.
- **Открывать Baileys-сокет из произвольного BullMQ-job'а.** Сокет открывает только процесс-владелец по Redis-реестру `instanceId → workerId`. Два процесса, держащие сокет на один `instanceId` = «conflict: replaced».
- **Держать Redlock-at-init как основной механизм удержания сокета** — short-lived лок с TTL не подходит для long-lived stateful ресурса. Только sticky ownership lifecycle.
- **Бесконечно авто-реконнектить `banned`-аккаунт.** Бан терминален; насиловать забаненный номер прокси/реконнектами запрещено.
- Вызывать WA `logout()` при обычном reconnect — только по `DisconnectReason.loggedOut` или явному действию пользователя. (Донор инвалидировал сессии и заставлял клиентов сканировать QR заново.)
- Доверять уникальности процесса через занятый порт. Использовать единого мастера/leader-election если несколько инстансов.
- Вставлять номера/тексты в SQL строкой.
- Передавать `access_token`/`instance_id` в URL query.
- Делать `COUNT(*)`/`GROUP BY` по живой `MessageLog` в пользовательских потоках — только `Stats`.

### WA-сессии
- Auth state — персистентное хранилище (Redis или БД), не только файлы на диске. (Донор страдал `Bad MAC` от рассинхрона `useMultiFileAuthState`.)
- Сериализовать рестарты сокета на один instance_id (один реконнект-маршрут), иначе «conflict: replaced».
- QR для уже зарегистрированной сессии не должен удалять креды.
- Watchdog сессий — с задержкой и одним проверяющим, не каждую секунду гонкой.

## 15. ИИ-агенты — особые правила

ИИ-агенты в этом проекте: ИИ-таргетолог (генерация креативов, запуск рекламных кампаний), чат-бот (квалификация лидов, ведение диалога), копирайтер (вариации текстов рассылок).

- **Детерминированная обёртка вокруг недетерминированного ядра.** Вызов LLM — за стабильным интерфейсом с типизированным входом/выходом (zod-схема ответа). Никогда не передавать raw LLM-ответ в мутацию без валидации.
- **Логирование и воспроизводимость.** Каждый вызов LLM логирует: prompt (хэш или id), модель, температуру, время, токен-usage, результат. Воспроизводимые тесты — на зафиксированных fixture-ответах (mock LLM), не на реальных вызовах.
- **Retry и fallback.** Временные сбои API → retry с backoff. Невалидный ответ → re-prompt с исправлением схемы, затем fallback на безопасное значение/человека.
- **Cost guard.** Оценка стоимости вызова до запуска; лимиты на токены/вызовы per tenant per day; жёсткие потолки для batch-операций (генерация креативов для N кампаний).
- **Safety rails.** Запретный контент-фильтр на входе и выходе. Никаких секретов/PII в prompt. Чат-бот не раскрывает системный промпт.
- **Человек в цикле для дорогих/рискованных действий.** Публикация рекламной кампании на реальный бюджет, отправка массовой рассылки, изменение тарифа — требуют подтверждения человеком. ИИ готовит, человек одобряет.
- **Тестируемость.** Чат-бот тестируется на каталоге сценариев диалогов (вход → ожидаемый переход/ответ). ИИ-таргетолог — на контракт: «на входе проект → на выходе валидный creative-payload, проходящий валидацию ad-API».
- **Не copy-paste промпт из донора.** Чат-бот донора по решению владельца не переносится — делаем свой, лучше.

## 16. UI And Design

Донор «ужасен в плане дизайна и структуры» — по словам владельца. Дизайн делаем с нуля, поэтому фиксируем дисциплину:

- Следовать дизайн-системе проекта (когда она появится — `packages/shared/ui` или внешний компонентный набор).
- Единый визуальный язык: типографика, отступы, радиусы, цвета, состояние.
- Отступы — на общей шкале (4/8/12/16/24/32...). Не one-off значения без визуального обоснования.
- Layout-ритм — через parent padding + container gap, не ad-hoc margin'ами.
- Shared-компоненты — визуально замкнутые единицы: surface, padding, radius, внутренние отступы, типографика, sizing контрола принадлежат компоненту.
- Компоновать shared-компоненты снаружи через wrapper'ы, не визуальными override'ами.
- Если потребителю нужно иное — сначала существующий semantic prop, затем минимальный новый semantic prop, затем локальный feature-wrapper.
- Не обходить established-примитивы ad-hoc поверхностями, если shared-примитив владеет этой ролью.
- Для frontend-багов инспектировать полный поток: маршрут → guard → layout → page → container → query → hook → handler → service → компонент → client contract → API → persistence.

## 17. Decision Rules

- Если решение очевидно и низкорисково — выполнять.
- Если есть материальные продуктовые/архитектурные tradeoff'ы — показать до двух жизнеспособных вариантов и порекомендовать один.
- Если безопасное предположение разблокирует работу — продолжить и указать предположение в финальном отчёте.
- Если действие деструктивное, необратимое, security/privacy-чувствительное или затрагивает чужих пользователей/данные — спросить перед выполнением.
- Если целевое поведение не достигнуто — сказать, что ещё не так, почему, и какой следующий эксперимент.

### Стоп-факторы (остановиться и спросить владельца)

- Смена WhatsApp-канала (Baileys ↔ Cloud API), смена платёжного провайдера, изменение тарифной сетки/лимитов, изменение публичных контрактов API.
- **Изменения схемы БД — только структурные/contract-breaking** (`drop`/`rename`/`type-change` существующих колонок, деструктивные операции) и **любые production-миграции**. Аддитивные dev-миграции (новые nullable-колонки/таблицы) идут **без остановки**, но фиксируются.
- Rename колонок проводить через expand-contract (добавить новое → бэкфилл → удалить старое), не «в лоб».
- Любая миграция, трогающая `teamId`/изоляцию тенантов, относится к стоп-классу (контрактное изменение по определению).

## 18. Completion Protocol

В конце каждой реализации или расследования отчитываться:

- что изменилось и почему;
- корневая причина, когда идентифицирована;
- затронутые слои;
- выполненная валидация;
- `Primary signal status`: met / not met / partially validated;
- `Secondary signal status`: какие именно проверки запущены и что показали;
- documentation status: обновлено / не нужно / требует выравнивания;
- оставшиеся риски, непокрытые места, follow-up;
- migration/rollout implications если контракты/схемы/персистентность/auth/маршрутизация/архитектура изменились;
- краткое предлагаемое commit-сообщение, когда изменение готово.

**Обязательное руководство по ручной проверке для пользователя (User Verification Guide):**
В каждом финальном ответе при передаче задачи на проверку ИИ-агент обязан предоставить чёткую, пошаговую инструкцию, чтобы пользователь мог самостоятельно и без лишних вопросов проверить готовую работу. Инструкция должна содержать:
1. Конкретные команды для запуска окружения и серверов (Docker, dev-сервер, миграции).
2. Точные URL-адреса, которые необходимо открыть.
3. Конкретные тестовые сценарии (какие данные ввести, на какие кнопки нажать).
4. Ожидаемый результат (как должен выглядеть интерфейс, какие изменения должны произойти в БД или логах).

Задача не выполнена, если видимый симптом ушёл, но тот же механизм остался структурно несогласованным по напрямую связанным слоям.

## 19. Production discipline (наследование от донора)

Эти процедуры оправдали себя в доноре и переносятся:

- Перед любым production-изменением — свежий backup (код + БД + WA-сессии + контрольные суммы).
- Правки — маленькими ветками и коммитами.
- Сначала staging (изолированный порт + отдельная БД + nulled-сессии + выключенные расписания/боты/вебхуки).
- На production — минимальный набор файлов.
- После deploy проверить: health endpoint, единственный активный процесс, порт, хвост логов, отсутствие массового QR/logout.
- Rollback должен быть готов до deploy.
- Не трогать production без явного согласия пользователя.
