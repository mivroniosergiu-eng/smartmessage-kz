# Дорожная карта Smartmessage.kz

Фазы разработки от фундамента до зрелой экосистемы. Каждая фаза: цель, скоуп, Definition of Done, привязка к плану (`../first plan.md`) и чекпоинтам (`QA_CHECKPOINTS.md`).

Принципы порядка фаз (почему именно так):
- **Фундамент раньше фич.** Без скелета монорепо, БД, очередей, auth и тест-инфраструктуры любая фича будет костыльной.
- **Этап 3 (рассылки) — самый сложный и рискованный**, но опирается на WA-слой, поэтому WA-подключение поднимаем рано (в Фазе 1), а массовую отправку — после CRM-ядра.
- **Этап 5 (биллинг) интегрируется последним**, но ограничения тарифа проектируются в схему с самого начала (`Permissions`), чтобы не переделывать.
- **Каждая фаза заканчивается пройденным ручным чекпоинтом** (`QA_CHECKPOINTS.md`) + зелёными автотестами (`TESTING.md`).

> Числа — оценки для планирования, не обязательства. Для сил ИИ-агентов это «порядок», не точные сроки.

---

## Протокол прохождения фазы (обязателен для ИИ-агента)

Каждая фаза проходится одинаково — через три гейта.

**1. Definition of Ready (вход).** Фазу можно начинать, только когда:

- предыдущая фаза в статусе Done (её выходные гейты пройдены);
- зависимости фазы доступны (см. блок «Зависимости» фазы);
- скоуп и Definition of Done понятны, а неоднозначные продуктовые решения сняты с владельцем (см. `AGENTS.md` §17 стоп-факторы).

**2. Рабочий цикл (TDD-first, см. `AGENTS.md` §4/§11).** Для каждой задачи фазы:

- самый ценный падающий тест → красный → минимальная реализация → зелёный → следующий случай;
- ослаблять тесты ради зелёного запрещено (anti-weakening);
- для денег/рассылок/auth/интеграций/ИИ — повышенные требования из `TESTING.md` (идемпотентность, retry, rate-limit, cross-tenant изоляция).

**3. Выходные гейты (Done).** Фаза не закрывается, пока не выполнены все три:

- ✅ автотесты зелёные локально (`pnpm build && pnpm test`) и в серверном CI; покрытие критичных зон не просело;
- ✅ пройден соответствующий раздел `QA_CHECKPOINTS.md` (ручной QA);
- ✅ заполнен отчёт в `docs/qa-runs/` (test-evidence ledger) со ссылкой на CI-run и результатами ручного QA.

Нет любого из трёх — фаза остаётся открытой.

## Фаза −1 — Спайк де-риска WhatsApp (1 неделя)

**Цель:** до написания сложного бэкенда на NestJS/BullMQ проверить пропускную способность, ban-rate и стабильность сессий Baileys на реальных номерах (см. `adr/0001-whatsapp-channel.md`). Донор `Ayat/smartmessage-waziper` уже подтверждает, что базовая рассылка через Baileys работает; спайк проверяет конкретные риски донора: повторный QR, `Bad MAC`, duplicate starts, reconnect/logout и потерю отправки при retry.

**Скоуп:**
- Изолированный «голый» скрипт Baileys на дешёвой VPS через прокси. В dev/spike допустим `useMultiFileAuthState`, потому что это baseline донора; результат должен показать, достаточно ли его для ограниченного запуска или нужен Redis/БД adapter уже в Фазе 1.
- Отправка контролируемой пачки сообщений с разным jitter; инжектированный ручной opt-out-текст для снижения жалоб.
- Замер реального **ban-rate**.
- Проверка сессионных сценариев: transient disconnect → reconnect без `logout()` и без повторного QR; рестарт процесса → auth-state жив; временный `session_error` → retry повторяет отправку того же сообщения; параллельный второй процесс не поднимает второй socket на тот же `instanceId`.

**⚠️ Критерии корректного замера (иначе цифры соврут):**
- тест на **прогретых** номерах (не свежих SIM — их забанят при любом jitter);
- получатели с **opt-in** / реалистичная база, не холодный спам;
- реалистичный контент сообщений.

**Definition of Done (DoD):**
- [ ] Порог успеха зафиксирован **до** эксперимента (пример: «ban-rate < X % на прогретых номерах с opt-in → канал жив, строим обвязку»; иначе — эскалация владельцу о миграции на Cloud API).
- [ ] Нет повторного QR после transient disconnect/restart для уже зарегистрированной сессии.
- [ ] Временный сбой отправки не теряет сообщение: reconnect/backoff → повторная отправка того же message-id.
- [ ] В логах нет socket-гонки `conflict: replaced` от двух владельцев одного `instanceId`.
- [ ] Результат и решение записаны в `docs/qa-runs/`; при необходимости обновлён `adr/0001-whatsapp-channel.md`.

## Фаза 0 — Фундамент и тест-инфраструктура

**Цель:** скелет, на котором можно безопасно строить.

**Скоуп:**
- pnpm workspace + монорепо по `ARCHITECTURE.md` §2 (`apps/web`, `apps/worker`, `packages/*`).
- `packages/db`: Prisma + PostgreSQL, начальная схема (`Team`, `User`, `WaAccount`, `WaSession`, `Contact`, `Campaign`, `MessageLog`, `Stats`, `Permissions`, `Subscription`-stub, `AuditLog`).
- `packages/shared`: контракты, zod-схемы, первые pure-утилиты (нормализация номеров, JID, spintax, `classifySendError`, `renderTemplate`).
- `packages/queue`: BullMQ-обёртка, test-Redis-конфиг, базовые очереди (заглушки).
- `apps/web`: Next.js с auth (регистрация/логин), защищённый маршрут, layout, дизайн-токены/общая шкала отступов.
- `apps/worker`: NestJS, health-endpoint, graceful shutdown.
- Тест-инфраструктура: Vitest, Playwright, test-БД, test-Redis, seed, `.env.example`.
- CI-agnostic скрипты: `pnpm lint/typecheck/test:*/build`.
- **Entitlements тарифов** в zod-схему (`packages/shared/src/tariffs.ts`, спека готова как DRAFT): лимиты пакетов ($200…$1000) известны коду с первого дня.
- **`prisma-erd-generator`**: ERD генерируется из `schema.prisma` автоматически при каждой миграции (ручной DATA_MODEL не заводим).
- **Quality-gate CI** (одобрено владельцем, см. `AGENTS.md` §11/§12): Husky pre-push (`pnpm build && pnpm test`) + серверный GitHub Actions с защитой от `.skip`/просадки покрытия (спека `.github/workflows/ci.yml` как DRAFT).
  - **Решение владельца в начале фазы:** `git init` + подключение GitHub-remote — обязательно для серверного CI-гейта (без репо гейт не работает). Агент не делает init сам — спрашивает.

**Definition of Done (DoD):**
- [ ] `AGENTS.md` §0 стек соблюдён; монорепо поднимается `pnpm install && pnpm dev`.
- [ ] Auth-поток: регистрация → логин → защищённая страница → разлогин (E2E зелёный).
- [ ] Все pure-утилиты `packages/shared` покрыты unit ≥ 90 %.
- [ ] test-БД мигрируется через `prisma migrate reset`; integration-тест пишет/читает.
- [ ] Воркер: graceful shutdown тест пройден (SIGTERM → текущие job'ы завершены).
- [ ] `QA_CHECKPOINTS.md` §0 пройден.
- [ ] В репо нет реальных секретов/номеров/сессий.
- [ ] `TierEntitlementsSchema` в `packages/shared` валидна; лимиты тарифов читаются из неё.
- [ ] `prisma-erd-generator` подключён; ERD генерируется при миграции.
- [ ] Husky pre-push активен; серверный CI зелёный и валит билд при `.skip`/`.only`/`xit` в диффе тестов или просадке покрытия.

## Фаза 4 — Массовые рассылки (Этап 3 плана) — критичная фаза

**Цель:** масштабная рассылка по базе с антибаном, устойчивостью, отчётностью. Самая рискованная зона.

**Скоуп:**
- `Campaign` flow: создание (draft) → старт (active) → пауза/возобновление → завершение.
- `packages/queue`: очередь `broadcast`, unique job id `(campaignId, contactId)`, идемпотентность.
- Троттлинг: per-кампания jitter (`minDelay/maxDelay`) + ротация аккаунтов + **глобальный rate-limit** на аккаунт (`REUSE_DONOR.md` §1.1).
- Retry с **повтором отправки** на `session_error`/`timeout` (`REUSE_DONOR.md` §2.4).
- Circuit-breaker: `technicalFailed >= N` → paused + уведомление.
- Spintax per-message + шаблонизатор `%key%`.
- Расписание: часовые окна, таймзоны, авто-completed.
- Статусы: `queued/sent/delivered/read/failed` + агрегация в `Stats`.
- Вебхук наружу (если настроен).
- `apps/web`: конструктор кампаний, мониторинг прогресса в реальном времени, отчёты, экспорт.

**DoD:**
- [ ] `QA_CHECKPOINTS.md` §3.2–§3.9 полностью пройдены.
- [ ] Идемпотентность: рестарт воркера mid-broadcast + повторный enqueue не дают дублей (`TESTING.md` §11).
- [ ] Retry повторяет отправку сообщения (защитный тест).
- [ ] Глобальный rate-limit соблюдается при параллельных кампаниях.
- [ ] Circuit-breaker срабатывает и уведомляет.
- [ ] Лимит тарифа глушит кампанию при превышении.
- [ ] Часовые окна и tz работают.
- [ ] Отчётность в реальном времени; экспорт работает.
- [ ] **Opt-out/отписка обязательна в каждом сообщении** + обработка отписки (прямая ban-митигация для Baileys: меньше жалоб → меньше блокировок).

## Зависимости между фазами

```
Фаза 0 (Фундамент)
  ├─→ Фаза 1 (WA-слой) ──┐
  ├─→ Фаза 2 (CRM-ядро) ─┤
  │                       ├─→ Фаза 3 (Чат-бот)
  │                       ├─→ Фаза 4 (Рассылки) ← зависит от WA + CRM
  │                       └─→ Фаза 5 (Лидогенерация) ← зависит от CRM
  ├─→ Фаза 6 (Обучение) ← независимо после Фазы 0
  └─→ Фаза 7 (Биллинг) ← независимо после Фазы 0, влияет на все через Permissions
       └─→ Фаза 8 (Prod readiness) ← после всех
```

Фазы 1, 2, 6, 7 могут идти параллельно после Фазы 0 (разные команды/агенты). Фазы 3, 4, 5 имеют зависимости и идут последовательно/с перекрытием по готовности зависимостей.
