# Ручные QA-чекпоинты Smartmessage.kz

Обязательные чек-листы, которые **человек** прогоняет перед продвижением фазы / релизом. Дополняют автотесты (`TESTING.md`): автотесты ловят регрессии, человек — то, что не выразить в коде (ощущение продукта, реальные внешние сервисы, кросс-системные сценарии, безопасность, edge-case'ы рассылок).

Принцип: **ни одна фаза не переходит в следующую, пока её чекпоинт не пройден и не зафиксирован** (кто, когда, результат, что найдено/исправлено).

> ⚠️ Чекпоинты с реальными внешними действиями (платные ad-кампании, реальная отправка в WhatsApp, реальные платежи) проводить на staging/isolated-аккаунтах. Production — только по согласованию.

---

## Как пользоваться

- Прогонять в браузере на свежем dev/staging-окружении с seeded-данными.
- Каждый пункт: отметить ✅ / ❌ / ⚠️ (частично) + краткий комментарий если ❌/⚠️.
- ❌ — блокер продвижения. ⚠️ — допустимо с зафиксированным риском и сроком исправления.
- Результаты сохранять в `docs/qa-runs/` (см. `docs/qa-runs/_TEMPLATE.md`) или в issue/PR фазы.

## Чекпоинт 1 — Этап «Трафик и ИИ-маркетинг»

Функционал: ИИ-таргетолог генерирует креативы → одобрение человеком → публикация в IG/FB/TikTok → лиды падают в CRM.

### 1.1. Генерация креативов
- [ ] ИИ-таргетолог на вход принимает описание проекта клиента и возвращает структурированный creative-payload (текст + визуал-мокап + аудитория).
- [ ] Результат LLM проходит валидацию zod-схемы. Невалидный ответ не роняет UI — есть понятная ошибка и повтор.
- [ ] Cost-guard виден: счётчик токенов/вызовов на tenant растёт; при превышении лимита — остановка с уведомлением, а не молчаливое превышение.
- [ ] В логе вызовов (`AiCallLog`) видны: агент, модель, токены, стоимость, статус.
- [ ] PII/секреты не попадают в prompt (визуально проверить в логе).
- [ ] Генерация для N кампаний (batch) не уходит в бесконечность — есть жёсткий потолок.

### 1.2. Одобрение человеком
- [ ] Креатив не публикуется автоматически — есть экран одобрения с возможностью reject/edit/approve.
- [ ] В `AuditLog` фиксируется, кто и когда одобрил.
- [ ] Отклонённый креатив не уходит в публикацию.

### 1.3. Публикация и лиды
- [ ] После approve вызывается ad-API (на test-аккаунте Meta/TikTok) — успех.
- [ ] Временный сбой ad-API → retry → понятное состояние «повтор», не silent loss.
- [ ] Перманентная ошибка ad-API → статус failed + видимое пользователю сообщение.
- [ ] Вебхук входящего лида (симулировать на staging) создаёт `Lead` и привязывает к команде.
- [ ] Дублирующий вебхук по тому же external-id не создаёт дубль-лида (идемпотентность).
- [ ] Лид появляется в CRM-списке с правильным `source`.

### 1.4. Безопасность и деньги
- [ ] Кнопка «Опубликовать на реальный бюджет» требует повторного подтверждения (человек в цикле).
- [ ] Бюджет кампании не может уйти в минус/превысить лимит без явного ввода.
- [ ] Токены ad-API берутся из env, не видны в UI-логах.

## Чекпоинт 2 — Этап «Ядро: CRM + чат-бот + воронка»

Функционал: лиды в воронке, чат-бот квалифицирует и ведёт диалог, календарь, менеджеры.

### 2.1. Воронка и CRM
- [ ] Лид движется по этапам `PipelineStage` вручную менеджером.
- [ ] Нельзя пропустить обязательный этап без подтверждения (если такие правила есть).
- [ ] Изменение этапа логируется и видно в истории лида.
- [ ] Фильтры/поиск по лидам работают (по менеджеру, этапу, источнику, дате).
- [ ] Пагинация/виртуализация не ломается на большом списке (seed 1000+ лидов).

### 2.2. Чат-бот
- [ ] Входящее сообщение (симулировать) → чат-бот отвечает по правилу (`keywords` + `typeSearch`).
- [ ] Exact-match и contains-режимы работают корректно.
- [ ] Default-бот срабатывает, когда ни одно правило не матчилось.
- [ ] Throttle/autoresponder delay: бот не спамит — отвечает не чаще, чем настроено.
- [ ] Чат-бот создаёт `CalendarItem` (созвон) по сценарию диалога.
- [ ] Уведомления о созвонах/встречах доходят (на staging-канале).
- [ ] Чат-бот не раскрывает системный промпт при прямом вопросе.
- [ ] Невалидный/нестандартный ответ LLM → fallback на безопасное сообщение, не падение.
- [ ] Handoff к менеджеру: бот корректно передаёт лид в ручной режим.

### 2.3. Менеджеры и права
- [ ] Менеджер видит только лидов своей команды (cross-tenant изоляция — попробовать чужой `leadId` в URL → 403/редирект).
- [ ] Роли owner/admin/manager дают ровно свои действия (например, только owner меняет тариф).
- [ ] Назначение лида на менеджера работает; фильтр «мои лиды» корректен.

### 2.4. Календарь
- [ ] Создание/редактирование/удаление событий работает.
- [ ] Таймзоны: событие в tz клиента отображается корректно в tz менеджера.
- [ ] Напоминания приходят вовремя (проверить на staging с коротким интервалом).

## Чекпоинт 3 — Этап «Дожим и массовые рассылки» (критичный)

Функционал: массовые WhatsApp-рассылки по базе, расписание, антибан, отчётность. Это самая рискованная зона.

### 3.1. Подключение WhatsApp-аккаунтов
- [x] Подключение WA-аккаунта через QR работает; статус → connected.
- [x] При транзиентном disconnect сессия переподключается **без** потери кредов и **без** `logout()` (QR не запрашивается заново).
- [x] `DisconnectReason.loggedOut` — единственный случай, когда требуется новый QR; креды очищаются безопасно.
- [x] Watchdog не устраивает гонок (нет «conflict: replaced» при ручном reconnect).
- [x] Несколько аккаунтов в одной команде изолированы по `instanceId`.

Пункты §3.1 закрыты 2026-07-22 в рамках Фазы 1: автоматические контракты и owner-authorized real-WA QR/restart smoke зафиксированы в `docs/qa-runs/2026-07-16-wa-real-qr-web.md` и `docs/qa-runs/2026-07-16-wa-single-send-web.md`.

### 3.2. Создание и запуск кампании
- [ ] Создание кампании (draft): выбор контактов, шаблон, расписание, задержки — сохраняется.
- [ ] Валидация: нельзя стартовать с пустым шаблоном / без аккаунтов / с некорректным расписанием.
- [ ] Старт → статус active → задания появляются в очереди (`MessageLog.status=queued`).
- [ ] Доступ по тарифу: превышение `whatsappMessagePerMonth` блокирует старт с понятной причиной.

### 3.3. Отправка и троттлинг (на isolated WA-аккаунте, малый объём)
- [ ] Сообщения уходят с задержкой в пределах `[minDelay, maxDelay]` — проверить по timestamp в `MessageLog`.
- [ ] Jitter: сообщения разносятся по времени, не залпом.
- [ ] Ротация аккаунтов: `nextAccount` переключается при нескольких аккаунтах.
- [ ] Global rate-limit: суммарная скорость по аккаунту ≤ лимита (проверить с 2 параллельными кампаниями).
- [ ] Spintax: получатели получают разные тексты для одного шаблона (проверить визуально).
- [ ] Presence/typing-имитация срабатывает перед отправкой.
- [ ] Нормализованный номер доходит (проверить на международном формате).

### 3.4. Идемпотентность и устойчивость — критично
- [ ] **Рестарт воркера mid-broadcast**: прогресс в БД не обнуляется; после рестарта — продолжение, дублей нет.
- [ ] **Повторный enqueue того же `(campaignId, contactId)`**: не создаёт дубль-отправки.
- [ ] **Retry при `session_error`/`timeout`**: то же сообщение отправляется повторно после реконнекта (не теряется).
- [ ] **Временный сбой Baileys** → классификация ошибки → retry с backoff.
- [ ] **Перманентная ошибка** (`media_error` и т.п.) → сообщение → failed с понятной причиной, кампания продолжается.

### 3.5. Circuit-breaker и лимиты
- [ ] При `>= N` technical_failed подряд кампания → auto-paused + уведомление пользователю.
- [ ] Лимит тарифа (`whatsappMessagePerMonth`) корректно глушит кампанию при превышении.
- [ ] Пауза/возобновление: задания останавливаются и продолжаются без дублей.

### 3.6. Расписание и часовые окна
- [ ] Кампания с `scheduleTime` (список часов) шлёт только в разрешённые часы.
- [ ] Если текущий час неразрешён — следующая отправка планируется на ближайший разрешённый.
- [ ] Таймзона кампании учитывается (создать кампанию в tz, отличной от серверной).
- [ ] Завершение: когда контакты исчерпаны → статус completed, `run` сброшен.

### 3.7. Отчётность и статусы
- [ ] `MessageLog.status` отражает queued/sent/delivered/read/failed.
- [ ] `Stats` (sent/failed/bulkTotal) корректно агрегируются.
- [ ] В UI виден прогресс кампании в реальном времени (обновление без перезагрузки).
- [ ] Экспорт/просмотр отчёта по конкретной кампании работает.

### 3.8. Валидация номеров
- [x] `onWhatsApp`-валидация ставит правильные статусы: confirmed / not_on_whatsapp / error.
- [x] Валидация идёт асинхронно, не блокирует UI.
- [ ] Невалидные номера исключаются из рассылки (или помечаются) по настройке.

Первые два пункта §3.8 закрыты 2026-07-22 в рамках Фазы 1; campaign gating невалидных номеров остаётся частью Фазы 3.

### 3.9. Входящие и чат-бот в контексте рассылки
- [ ] Ответ клиента на рассылку обрабатывается (webhook/`messages.upsert`).
- [ ] Autoresponder не спамит при повторных ответах (throttle по `lastResponse`).
- [ ] Внешний webhook наружу (если настроен) получает корректный payload.

## Когда чекпоинты НЕ заменяют автотесты

- Ручные проверки не дают права убрать автотест на то же поведение. Чекпоинт = дополнительный слой перед фазой/релизом.
- Если чекпоинт регулярно ловит то, чего нет в автотестах, — добавить автотест (не делать ручную проверку постоянной).
- Если ручной чекпоинт невозможно воспроизвести детерминированно (например, реальный бан WA), он остаётся ручным, но фиксируется максимально формально.
