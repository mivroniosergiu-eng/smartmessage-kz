# ADR-0002. Платёжный провайдер (Merchant of Record)

Статус: принято. Дата: 2026-06-30.

## Контекст
Stripe официально недоступен для онбординга мерчантов в Казахстане. Кодить его как основное решение — значит писать код в стол.

## Решение
Первой и единственной целевой реализацией биллинга пишется **Merchant of Record (Paddle или Lemon Squeezy)** — они берут на себя налоги, инвойсы и работают глобально, что для SaaS из РК на $-тарифах оптимально.

- В коде — **тонкий интерфейс `PaymentService`**, единственная реализация пока MoR. Фабрику провайдеров **не строим** (оверинжиниринг).
- Финальный выбор Paddle vs Lemon Squeezy делается отдельной задачей до начала Фазы 7.
- **Контракт Checkout-сессии** обязан нести `metadata: { teamId, subscriptionId }`; webhook **идемпотентен** по event-id и привязывает оплату к тенанту через эту metadata.
- В схеме БД: уникальные индексы (`@unique`) на `providerSubId` и `providerCustomerId`. Рядом — поле `paymentProvider` (какой именно MoR), иначе `providerSubId` без указания провайдера неоднозначен.

## Последствия
Имена провайдер-нейтральные (`providerSubId`, а не `stripeSubId`) во всех файлах и доках. Смена провайдера в будущем = новая реализация за `PaymentService` без изменения доменной модели.
